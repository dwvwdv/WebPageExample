字體反爬

反爬具體思路

使用@font-face，將字體檔各類編碼藏入後用對應關係呈現，沒有對應字型檔無法呈現網頁內容，以實現反爬，將不同編碼位置硬性塞入根本不屬於他的字元，

例如 U+98A8是風的unicode編碼，這時我將U+2605的字體改成風，並將網頁上所有呈現內容裡有風的全部替換成U+2605編碼的風，就可以成功反制爬蟲，爬蟲在解析這個文字的時候，會因為缺少對應字體檔案而無法正確解析顯示，於是會變成解析那個位置本來的unicode，也就是U+2605(這個編碼就是★)，只要有心去做，甚至能將網頁上內容爬下來後，內容是完全不一致，且能做到爬下來的內容也是有意義的

舉個例子，網頁上內容呈現

今天太陽好大呀

這是完全可以根據unicode量身訂做字體的，在爬取下來後變成

昨日夜色太美了

但耗費的時間、心力也是極大

@font-face{
	font-family: fontName,
	src: local("HelveticaNeue-Bold"),
			url(/fontsURL/font.woff);
}

font-family為font-face在CSS的名字

在src標籤中的url給予字體檔位置，有多個相同檔、不同副檔名是為了兼容瀏覽器

字形格式

  • eot

    微軟創造字形格式，IE6-IE8使用

  • woff

    載入速度通常快些，且現代所有瀏覽器皆支援

  • otf/ttf

    易複製、容易傳播開

  • svg/svgz

    體積比向量圖小

反反爬

在處理CSS加密時，第一時間就是追蹤解析不出文字的CSS，由於大同小異，只解析其中一個，畢竟處理過反爬的網站，爬蟲難度總是大幅增高，尤其在作報告時網站好像又新增了一層驗證，所以我後面在爬都爬不太到，只能抓請求封包模擬成chrome瀏覽器訪問的封包，總之非常繁瑣，我們得到一個CSS class-shopNum，打開後發現是一堆字體檔，將這些字體檔全部下載下來，一一解析後發現，每個檔案都是603字，所有字體一模一樣，但全被塞入不同的unicode位置，想要完全解析就得將unicode和他被改造的一一對上，全部只能手動自己key進字典，再將Response的content裡有unicode編碼解析不出來的一一還原。

後來我想想，我們可以換一個角度來看這個字體加密，假設說今天網頁上內容是

哈哈哈哈哈哈哈哈

爬取下來後卻顯示

這是一個秘密訊息

這代表網頁這八個哈，是8個完全不同unicode編碼，是不是也做到了反向的CSS加密，這是只有將網頁與字體檔分開來，才能做到的解密。
